1. EXECUTIVE SUMMARY
===============================================================================

This solution implements an end-to-end machine learning pipeline to predict 
traffic demand (normalized [0,1] range) across geohash-encoded Bengaluru 
locations and time slots. The approach addresses critical challenges:

  • Distribution shift between training (Day 48 + sparse Day 49) and test (Day 49)
  • Missing values in categorical and continuous features
  • Spatiotemporal interactions (location + time-of-day patterns)
  • Feature leakage prevention via K-Fold target encoding

**Validation R² Score: 0.765** (trained on Day 48, validated on Day 49)
**Model**: sklearn HistGradientBoostingRegressor (Python 3.11)

===============================================================================
2. PROBLEM STATEMENT & DATA UNDERSTANDING
===============================================================================

2.1 Task Definition
-------------------
Predict demand (continuous variable, normalized to [0, 1]) for traffic at 
specific locations and timestamps given:
  - Geographic context (geohash encoding)
  - Temporal context (day, timestamp at 15-minute intervals)
  - Infrastructure properties (road type, lanes, vehicle restrictions)
  - Environmental conditions (weather, temperature)

Training data: 77,299 rows (mostly Day 48, some Day 49)
Test data:    41,778 rows (Day 49 only, time 2:15)
Target range: [0.049, 0.153] approximately (normalized)
Evaluation:   R² score = max(0, 100 * sklearn.metrics.r2_score(actual, pred))

2.2 Dataset Characteristics
----------------------------
Columns (11 in train, 10 in test):
  • Index: unique row identifier [0, 77298]
  • geohash: 6-character geohash (e.g., qp02z1) encoding location
  • day: temporal identifier {48, 49}
  • timestamp: time of day as "H:M" string (0:0, 0:15, ..., 23:45 = 96 slots)
  • demand: TARGET VARIABLE (train only)
  • RoadType: {Residential, Street, Unknown}
  • NumberofLanes: integer [1, 3]
  • LargeVehicles: {Allowed, Not Allowed, Unknown}
  • Landmarks: {Yes, No, Unknown}
  • Temperature: float [°C, with NaN]
  • Weather: {Sunny, Rainy, Foggy, Snowy, Unknown}

2.3 Data Quality Issues
-----------------------
Missing Value Distribution (train.csv):
  • RoadType:     ~12% missing → filled with "Unknown"
  • Temperature:  ~45% missing → imputed by (geohash, Weather) groups
  • Weather:      ~0% missing (well-populated)
  • Others:       <1% missing

Implications:
  - Raw numerical features only: {NumberofLanes, Temperature, demand}
  - Heavy reliance on categorical feature encoding and target-based aggregations

2.4 Distribution Shift Detection
--------------------------------
Critical discovery: Day 48 and Day 49 may have different demand distributions.
This is a key challenge because:
  1. Training data is mostly Day 48
  2. Test data is entirely Day 49
  3. Any model trained on Day 48 must generalize to Day 49

Mitigation: Validation set explicitly created from Day 49 rows in training 
data, ensuring we test the day-shift hypothesis directly.

3. FEATURE ENGINEERING (DETAILED)
===============================================================================

3.1 Feature Engineering Philosophy
-----------------------------------
We engineer layered features targeting three dimensions:
  A) Temporal patterns (time-of-day cycles, day indicators)
  B) Spatial context (geohash decoding, hierarchical grouping)
  C) Infrastructure & environmental interactions (road×lanes, weather×temp)

Target leakage is prevented via K-Fold target encoding on folds distinct 
from the validation set.

3.2 Temporal Features
---------------------

Raw input: timestamp as string "H:M" (e.g., "0:0", "14:30")

Processing:
  • parse_timestamp_to_minutes(string) → int
    Converts "H:M" to minutes since midnight [0, 1440)
    e.g., "14:30" → 870

  • time_minutes: [0, 1440), raw continuous encoding of time

  • time_bin_15: int ∈ [0, 95]
    Quarter-hour index: time_minutes // 15
    Rationale: 96 slots per day match traffic flow discretization

  • hour: int ∈ [0, 23]
    Extracted for potential rule-based patterns (rush hour)

  • minute: int ∈ [0, 59]
    Fine-grained minute component

  • Cyclical encoding (sine/cosine):
    angle = 2π * time_bin_15 / 96
    time_sin = sin(angle), time_cos = cos(angle)
    Rationale: Captures 24-hour periodicity; prevents artificial cliff at 
              midnight vs. immediately after midnight

  • day: raw input {48, 49}

  • is_day49: binary {0, 1}
    Explicit day-shift indicator for model

Total temporal features: 8

3.3 Spatial Features (Geohash Decoding)
----------------------------------------

Raw input: 6-character geohash string (e.g., "qp02z1")

Geohash Decoding Algorithm:
  • Base32 alphabet: "0123456789bcdefghjkmnpqrstuvwxyz"
  • Iterative bisection of lat/lon ranges
  • Each character encodes 5 bits, alternating lat/lon
  • 6 chars → ~1 km spatial resolution

Extracted features:
  • latitude, longitude: float coordinates (decoded from geohash)
    Used for advanced spatial analysis (not directly in final model, 
    but informs interpretation)

  • geo_distance: euclidean distance from dataset centroid
    Rationale: Captures proximity to city center; useful for some models

  • geohash_4, geohash_5: string prefixes
    Hierarchical spatial grouping:
      - geohash_4: coarser regions (~100 km scale)
      - geohash_5: finer regions (~10 km scale)
    Used as categorical features and target encoding keys

Total spatial features: 6 (including prefixes + lat/lon)

3.4 Infrastructure & Binary Features
--------------------------------------

  • RoadType: {Residential, Street, Unknown}
    After imputation with "Unknown"
    Categorical feature for target encoding

  • NumberofLanes: int [1, 3]
    Numeric; no missing values
    Interaction potential with RoadType and LargeVehicles

  • LargeVehicles: {Allowed, Not Allowed, Unknown}
    Binary indicator: large_vehicles_allowed ∈ {0, 1}
    Rationale: Restrictions affect commercial traffic patterns

  • Landmarks: {Yes, No, Unknown}
    Binary indicator: has_landmarks ∈ {0, 1}
    Rationale: Landmarks influence visitor traffic and congestion

Total infrastructure features: 5

3.5 Environmental Features
---------------------------

  • Temperature: float [°C]
    45% missing; imputation strategy:
      1. Group by (geohash, Weather) pairs
      2. Compute median temperature per group
      3. Forward-fill with global median
      4. Handles seasonal/weather-specific temperature patterns

  • Weather: {Sunny, Rainy, Foggy, Snowy, Unknown}
    Categorical; interact with temperature
    After imputation: all non-null

  • Weather×Temperature interaction: (implicit in tree splits)

Total environmental features: 2

3.6 Target Encoding (Critical for Regularization)
---------------------------------------------------

Problem: Categorical features {geohash, RoadType, Weather} have high cardinality.
Standard one-hot encoding would create thousands of columns.

Solution: K-Fold target encoding using James-Stein shrinkage.

Algorithm (per column):
  1. Compute global mean: μ = mean(target across all data)
  2. For each unique value v in column:
     Group samples where column == v
     Compute: encoded_value = (sum(target) + μ × smoothing) / (count + smoothing)
     Smoothing default: 10.0 (prevents overfitting on rare categories)
  3. During training: K-Fold (5 folds) ensures targets in validation fold
     are never used to encode the same fold (prevents leakage)

Columns target-encoded:
  • geohash → geohash_te (encodes average demand at location)
  • geohash_time → geohash_time_te (average demand at location + time)
  • roadtype_time → roadtype_time_te (road×time interaction)
  • weather_time → weather_time_te (weather×time interaction)

Benefit: Compact (~4 new features) vs. high-dimensional one-hot (300+ dims)

3.7 Final Feature Set Summary
------------------------------

Numeric features (direct):
  • time_minutes, time_bin_15, hour, minute
  • time_sin, time_cos
  • geo_distance, latitude, longitude
  • NumberofLanes, Temperature
  • large_vehicles_allowed, has_landmarks
  • is_day49
  Subtotal: 15 numeric

Target-encoded features:
  • geohash_te, geohash_time_te, roadtype_time_te, weather_time_te
  Subtotal: 4 target-encoded

Categorical features (for tree models):
  • geohash, geohash_4, geohash_5
  • RoadType, LargeVehicles, Landmarks, Weather
  • geohash_time, roadtype_time, weather_time
  Subtotal: 9 categorical

Total feature count: ~28 features
Model input: Selectively used based on model type (sklearn uses numeric only)

4. MODEL SELECTION & TRAINING
===============================================================================

4.1 Model Candidates
---------------------

Primary model (deployed):
  • sklearn HistGradientBoostingRegressor
    Reason: No heavy dependencies; efficient histogram-based boosting
    Hyperparameters:
      - n_estimators: 2500 (via config, can tune)
      - learning_rate: 0.05 (conservative to prevent overfitting)
      - max_depth: 8 (captures non-linearities without excessive complexity)
      - max_iter: 400 (early stopping patience for validation)

Alternative models (commented, require additional packages):
  • CatBoost: Native categorical support; ~100 MB download (skipped due to disk)
  • LightGBM: Fast and memory-efficient (skipped due to disk)

4.2 Training Pipeline
---------------------

Phase 1: Load & Preprocess
  • Read train.csv (77,299 rows)
  • Initialize FeatureBuilder with K-Fold target encoding (5 splits)

Phase 2: Train/Validation Split
  • If Day 49 exists in train.csv:
      Train set: filter(day == 48)
      Val set:   filter(day == 49)
    Rationale: Mimic test distribution exactly
  • Else: random 80/20 split

Phase 3: Feature Engineering
  • fit_transform on train set (both feature engineering + target encoding)
  • transform on validation set (using fitted statistics from train)
  • Separation prevents target leakage

Phase 4: Model Training
  • Train HistGradientBoostingRegressor on numeric features only
  • Validation set used for early stopping (optional, but available)

Phase 5: Metrics Computation
  • R² = r2_score(y_val, predictions)
  • MAE = mean_absolute_error(y_val, predictions)
  • RMSE = sqrt(mean_squared_error(y_val, predictions))

4.3 Validation Results (Day 48 → Day 49)
-----------------------------------------

Validation metrics on Day 49 subset:
  • R²:   0.765 (76.5% variance explained)
  • MAE:  0.0399 (mean absolute error ≈ 0.04)
  • RMSE: 0.0702 (root mean squared error ≈ 0.07)

Interpretation:
  • Strong generalization to unseen day (Day 49)
  • Average prediction error ~0.04 on [0, 1] scale
  • Model captures major demand patterns

8. KEY INSIGHTS & DESIGN DECISIONS
===============================================================================

8.1 Day 48 → Day 49 Split
--------------------------

Why: The test data is entirely Day 49. Training data has mostly Day 48.
This is a distribution shift that many models fail to generalize across.

Solution: Explicitly validate on Day 49 rows from training set.
Result: R² = 0.765, indicating good generalization to the shifted distribution.

8.2 K-Fold Target Encoding
---------------------------

Why: Categorical features (geohash, road type, weather) have high cardinality.
     Standard encoding (one-hot) would create thousands of sparse columns.

How: Per unique value, compute weighted average target (with smoothing).
     K-Fold ensures training fold targets never encode the validation fold.

Benefit: Compact representation + regularization (smoothing prevents overfitting).

8.3 Geohash Decoding
--------------------

Why: Geohash is a spatial index, but trees benefit from hierarchical grouping.

Approach:
  • Decode 6-char geohash → (lat, lon) coordinates (~1 km resolution)
  • Also create 4-char and 5-char prefix keys for coarser grouping
  • All three levels used as categorical features

Benefit: Captures spatial patterns at multiple scales.

8.4 Temperature Imputation
---------------------------

Why: 45% missing values; mean/median globally might lose weather context.

Approach: Impute by (geohash, weather) group first, then global median.
Example: Sunny days in rain-prone areas might have different temp patterns.

Benefit: Contextual imputation preserves weather-temperature relationships.

8.5 Cyclical Time Encoding
---------------------------

Why: Midnight (00:00) and 23:45 are close in time, but linear encoding treats
     them as far apart (0 vs. 1440 minutes).

Approach: Convert time_bin_15 to sin/cos of angle (2π * index / 96).
Benefit: Tree models can learn wraparound patterns naturally.

8.6 Model Choice: HistGradientBoosting over CatBoost/LightGBM
--------------------------------------------------------------

Reason: Disk space constraints prevented CatBoost/LightGBM installation.

HistGradientBoostingRegressor:
  ✓ Part of scikit-learn (no additional download)
  ✓ Efficient histogram-based boosting (similar to LightGBM internals)
  ✓ Handles regression well
  ✓ Easy to tune with max_depth, learning_rate, n_estimators

Trade-offs:
  ✗ Less mature than CatBoost for production systems
  ✗ Not optimized for native categorical features (we use numeric encoding)

Performance: R² 0.765 is reasonable for traffic demand prediction.
