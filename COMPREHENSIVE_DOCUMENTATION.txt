3.1 Feature Engineering Philosophy & Architecture
---------------------------------------------------

The feature engineering pipeline is built on three core principles:

1. TEMPORAL DIMENSIONALITY
   - Capture cyclical patterns (24-hour periodicity)
   - Encode time granularity (15-minute intervals)
   - Enable time-of-day interaction with other features
   - Use sin/cos transforms for cyclical encoding (prevents boundary artifacts)

2. SPATIAL REASONING
   - Decode geohash to lat/lon coordinates
   - Extract hierarchical geographic groupings (1km, 5km, 10km scales)
   - Enable location-based aggregations (geohash-level demand patterns)
   - Preserve geographic continuity through encoded representation

3. CONTEXTUAL INTERACTIONS
   - Model infrastructure × traffic relationships
   - Capture weather-dependent traffic variations
   - Encode time-dependent road characteristics
   - Enable non-linear feature combinations through XGBoost/HistGradient

Feature Implementation Overview:
  Raw Input: 11 columns (index + 10 features)
  Engineered: 28 columns (after transformations)
  Final Model Input: All 28 features
  Feature Selection: None (all engineered features retained)

3.2 Temporal Feature Engineering (Detailed)
---------------------------------------------

Raw Input: timestamp string "H:M" (e.g., "14:30")

Step 1: Parse Timestamp to Minutes
  - Convert "H:M" format to integer minutes since midnight
  - "0:0" → 0 minutes
  - "0:15" → 15 minutes
  - "14:30" → 870 minutes
  - "23:45" → 1425 minutes
  - Range: [0, 1425] (1440 = 24*60)
  - Algorithm: hour * 60 + minute
  - Validation: 0 <= hour <= 23, 0 <= minute <= 59

Step 2: Convert Minutes to Quarter-Hour Index
  - Divide minutes by 15 to get quarter-hour slot
  - time_slot = minutes // 15
  - Range: [0, 95] (96 slots = 24 hours / 0.25 hour)
  - Represents which 15-minute period within 24-hour day
  - Example: 870 minutes // 15 = 58 (9:30 AM slot)

Step 3: Cyclical Time Encoding (Sin/Cos Transform)
  - Problem: Time is cyclical (23:45 is 15 min before 0:0, not 12+ hours away)
  - Naive one-hot encoding: Creates discontinuity at midnight
  - Solution: Sine/Cosine encoding (circular coordinate representation)
  
  Formula:
    - phase = 2 * π * (time_slot / 96)  [converts slot to radians, period = 96]
    - time_sin = sin(phase)               [range: [-1, 1]]
    - time_cos = cos(phase)               [range: [-1, 1]]
  
  Properties:
    - 0:0 (slot 0) → sin≈0, cos≈1
    - 6:0 (slot 24) → sin≈1, cos≈0
    - 12:0 (slot 48) → sin≈0, cos≈-1
    - 18:0 (slot 72) → sin≈-1, cos≈0
    - 23:45 (slot 95) → sin≈-1, cos≈1 (close to 0:0)
  - Captures periodicity naturally
  - Euclidean distance respects circular nature

Step 4: Time-of-Day Category Encoding
  - Create categorical "time_of_day" based on demand patterns:
    * "early_morning" (0:00 - 5:59) - slots 0-23
    * "morning_commute" (6:00 - 9:59) - slots 24-39
    * "daytime" (10:00 - 16:59) - slots 40-67
    * "evening_commute" (17:00 - 20:59) - slots 68-83
    * "night" (21:00 - 23:59) - slots 84-95
  - Reflects expected traffic patterns in Bengaluru
  - One-hot encoded (5 binary features) or label encoded
  - Enables explicit time-period interactions

Step 5: Day-Based Features
  - Extract raw day value: {48, 49}
  - Create binary day_is_49 feature: 1 if day == 49, 0 otherwise
  - Purpose: Explicitly capture Day 48 vs Day 49 distribution differences
  - Rationale: Day-based features help model learn day-specific patterns

Generated Temporal Features:
  - time_minutes: [0, 1425]
  - time_slot: [0, 95]
  - time_sin: [-1, 1]
  - time_cos: [-1, 1]
  - time_of_day: categorical {early_morning, ..., night}
  - day_is_49: binary {0, 1}

3.3 Spatial Feature Engineering (Detailed)
--------------------------------------------

Raw Input: geohash string "qp02z1" (6 characters, base32 encoded)

Step 1: Geohash Structure Understanding
  - Format: 6 characters, each character = 5 bits (base32)
  - Total: 30 bits precision
  - Encoding: Alternates between longitude and latitude bits
  - Resolution: ~1 km² per 6-character geohash in Bengaluru region
  - Hierarchical: First N characters represent coarser resolution
    * 2 chars: ~100 km² (state level)
    * 3 chars: ~20 km² (city level)
    * 4 chars: ~5 km² (district level)
    * 5 chars: ~1.2 km² (neighborhood level)
    * 6 chars: ~0.15 km² (block level)

Step 2: Decode Geohash to Latitude/Longitude
  - Algorithm: Binary search in lat/lon space
  - For each character (base32 → 5 bits):
    * Iterate through 5 bit masks: {16, 8, 4, 2, 1}
    * Alternate between longitude (even) and latitude (odd)
    * If bit set: narrow upper half; if unset: narrow lower half
  - Result: Approximate lat/lon center of geohash cell
  
  Example: "qp02z1"
    - q = 16 (base32) = 10000 (binary)
    - p = 15 (base32) = 01111 (binary)
    - Iterates through bit masks to narrow lat/lon ranges
    - Outputs: (lat ≈ 12.95°, lon ≈ 77.58°) [Bengaluru]
  
  Generated Features:
    - latitude: [-90, 90]
    - longitude: [-180, 180]

Step 3: Create Hierarchical Geohash Features
  - Geohash_2: First 2 characters (100 km² regions)
  - Geohash_3: First 3 characters (20 km² regions)
  - Geohash_4: First 4 characters (5 km² regions)
  - Geohash_5: First 5 characters (1.2 km² regions)
  - Geohash (full): All 6 characters (exact location)
  
  Purpose: Hierarchical aggregation levels
    * 6-char: Very specific locations (few samples, high variance)
    * 5-char: Neighborhoods (moderate aggregation)
    * 4-char: Districts (coarse aggregation, stable estimates)
    * 2-char: City regions (very coarse, interpretable)
  
  Target Encoding Strategy:
    * Use full geohash_6 in target encoding
    * Use geohash_4, geohash_5 as categorical features
    * Hierarchical target encoding: Learn demand patterns at multiple scales

Generated Spatial Features:
  - latitude, longitude: Continuous [float]
  - geohash_2, geohash_3, geohash_4, geohash_5: Categorical
  - geohash (full): Categorical (high cardinality ~200-300 unique)

3.6 Target Encoding Implementation (Advanced)
-----------------------------------------------

Purpose: Convert high-cardinality categorical features into continuous values
using target statistics (mean demand per category).

Problem: Direct groupby().mean() causes target leakage (model sees training
target when encoding training data).

Solution: K-Fold Cross-Validation Target Encoding

Algorithm:
  
  For each feature column in TARGET_ENCODER_COLUMNS:
    1. Split data into K=5 folds (stratified by day)
    2. For each fold i:
       - Training folds: 1,2,3,4 (if fold 5 is validation)
       - Validation fold: 5
       - Compute: mapping[value] = (sum_target_in_train + λ*global_mean) / (count_in_train + λ)
         where λ=10.0 (smoothing parameter) prevents overfitting rare categories
       - Apply: validation_fold[feature_te] = mapping[corresponding_values]
    3. Repeat for all 5 folds
    4. Result: Out-of-fold (OOF) encoded feature (no leakage)
    5. Global fit: Fit encoder on entire dataset for test prediction

Smoothing Parameter (λ = 10.0):
  - Balances global mean vs group mean
  - Raw estimate: sum_target / count
  - Smoothed: (sum_target + λ*global_mean) / (count + λ)
  - If count is large: Converges to group mean
  - If count is small: Defaults toward global mean
  - Prevents extreme values for rare categories

Columns Encoded:
  - geohash (6-char, ~300 unique) → geohash_te
  - geohash_time (interaction, ~1500 unique) → geohash_time_te
  - roadtype_time (interaction, 15 unique) → roadtype_time_te
  - weather_time (interaction, 25 unique) → weather_time_te

Output: 4 new continuous features with range roughly equal to target range

4.1 Model Choice: sklearn HistGradientBoostingRegressor
------------------------------------------------------

Why HistGradientBoosting?

Advantages Over Alternatives:

  vs. Linear Models (Linear Regression, Ridge, Lasso):
    - Demand has non-linear spatiotemporal patterns (peaks at specific times)
    - Linear models insufficient for capturing interactions
    - HistGradient provides automatic interaction learning

  vs. Basic Decision Trees:
    - Single tree: High variance, overfits
    - HistGradient: Ensemble of trees, reduces variance
    - Boosting: Sequential error correction improves accuracy

  vs. Random Forests:
    - RF: Parallel trees, limited sequential error correction
    - HistGradient: Boosting + histogram binning for speed
    - HistGradient: Better for tabular data, especially with many features

  vs. CatBoost / LightGBM:
    - All three are similar (histogram-based boosting)
    - HistGradient: Pure sklearn (no external dependency)
    - CatBoost: Better categorical handling (but slower)
    - LightGBM: Fastest, but sklearn native categorical support better
    - Choice: HistGradient for simplicity + dependency minimalism

  vs. Neural Networks:
    - Small dataset size (77k rows, 28 features)
    - Tree models dominate tabular data (Kaggle benchmarks confirm)
    - Neural networks need more data to justify overhead
    - Training time: Trees << Neural nets

Strengths for This Problem:
  1. Captures non-linear spatiotemporal interactions naturally
  2. Handles mixed feature types (categorical + continuous)
  3. Native categorical support (no one-hot encoding needed)
  4. Insensitive to feature scaling
  5. Interpretable feature importances
  6. Fast training & prediction (histogram-based)
  7. No external C++ dependencies (pure Python + NumPy)
  8. Robust to outliers via histogram binning

Weaknesses:
  1. Less interpretable than linear models
  2. Black-box nature (feature interactions implicit)
  3. Requires careful hyperparameter tuning
  4. Prone to overfitting if not regularized

4.2 Hyperparameter Configuration
---------------------------------

Key Hyperparameters:

learning_rate = 0.05
  - Shrinkage applied to each tree's contribution
  - Formula: updated_prediction = old_prediction + learning_rate * tree_prediction
  - Low rate (0.05): Smaller steps, less overfit, more stable
  - Trade-off: Need more iterations to converge
  - Chosen: 0.05 (conservative, stable)

max_depth = 8
  - Maximum tree depth (number of levels)
  - Depth 1: Single split (stumps)
  - Depth 8: Up to 2^8 = 256 leaf nodes per tree
  - Higher depth: More complex interactions, higher overfit risk
  - Chosen: 8 (moderate complexity, prevents excessive variance)

max_iter = 400 (sklearn's n_estimators equivalent)
  - Number of boosting iterations
  - Each iteration: Fit one tree to residuals
  - Effect: learning_rate * max_iter determines total shrinkage
  - Higher: Better training accuracy, overfit risk
  - Chosen: 400 (sufficient for convergence, ~2-3 hours training)

l2_regularization = 0.0
  - L2 penalty on tree leaves
  - Defaults to 1.0 (shrinks leaf values)
  - Set to 0.0: No regularization (let data speak)
  - Alternative: 0.1-1.0 if overfitting detected
  - Chosen: 0.0 (data is regularized via learning_rate + max_depth)

random_state = 42
  - Seed for reproducibility
  - Ensures same results across runs
  - Needed for comparing hyperparameter changes

Complete Configuration (YAML):
```yaml
learning_rate: 0.05
max_depth: 8
max_iter: 400
l2_regularization: 0.0
random_state: 42
```

    * Temperature → (geohash, weather) grouped median
    * LargeVehicles, Landmarks → "Unknown"
  - Validate: No NaN remaining in features
  - Output: Clean dataframes

Step 3: Train-Validation Split
  - Strategy: Day-based split (if Day 49 exists in train)
    * train_df: day == 48 (70,000+ samples)
    * val_df: day == 49 (1,200+ samples)
  - Rationale: Mimics distribution shift in test set
  - If Day 49 unavailable: Random 80/20 split
  - Output: Separate training and validation frames

Step 4: Feature Engineering (FeatureBuilder Class)
  - Create temporal features: time_sin, time_cos, time_slot, time_of_day
  - Decode geohash: latitude, longitude
  - Create hierarchical geohashes: geohash_2, geohash_3, etc.
  - Create interaction features: roadtype_time, weather_time, geohash_time
  - Encode raw categorical features
  - Output: ~25-28 raw features (before target encoding)

Step 5: Target Encoding (K-Fold)
  - Fit TargetEncoder on train_df
  - Transform train_df → training OOF encoded features
  - Transform val_df → validation encoded features
  - Store encoder for test prediction
  - Output: Target-encoded features added to train/val dataframes

Step 6: Final Feature Selection
  - Select all 28 features for model input
  - No feature selection/elimination applied
  - Rationale: All features have interpretable meaning
  - Output: X_train (70k × 28), y_train (70k,), X_val (1.2k × 28), y_val (1.2k,)

Step 7: Model Training
  - Initialize: HistGradientBoostingRegressor(hyperparams)
  - Fit: X_train, y_train
  - Time: ~10 minutes (CPU, no GPU)
  - Output: Trained model object

Step 8: Model Evaluation
  - Predict on validation: y_val_pred = model.predict(X_val)
  - Compute metrics:
    * R² = r2_score(y_val, y_val_pred) = 0.765
    * MAE = mean_absolute_error(y_val, y_val_pred) = 0.0399
    * RMSE = sqrt(mean_squared_error(y_val, y_val_pred)) = 0.0702
  - Output: Validation metrics (printed to console)

Step 9: Full-Data Model Training (for Submission)
  - Combine train + val data (entire Day 48 + Day 49)
  - Re-run feature engineering + target encoding on combined data
  - Train new model on all 77k samples
  - Output: Full model (slightly better generalization)

Step 10: Test Set Prediction
  - Load test.csv
  - Apply same preprocessing (fill missing values identically)
  - Apply same feature engineering (using fitted FeatureBuilder)
  - Apply same target encoding (using fitted TargetEncoder)
  - Predict: test_predictions = model.predict(X_test)
  - Output: 41,778 normalized demand predictions

Step 11: Submission Generation
  - Create submission DataFrame:
    * Column 'Index': [0, 1, ..., 41777]
    * Column 'demand': [pred_0, pred_1, ..., pred_41777]
  - Save to CSV: outputs/submission.csv
  - Output: Ready-to-submit file

5.2 Validation Strategy Details
--------------------------------

Why Day 48 vs Day 49 Split?

  Training Distribution:
    - 76,000+ samples from Day 48
    - ~1,200+ samples from Day 49
    - Aggregate: Model sees mostly Day 48
  
  Test Distribution:
    - 41,778 samples from Day 49
    - Test is 100% Day 49
  
  Hidden Challenge:
    - Day 48 and Day 49 have different demand patterns
    - Weather, events, congestion different between days
    - Model trained on Day 48 may fail on Day 49 (distribution shift)
  
  Validation Approach:
    - Train exclusively on Day 48 (70k samples)
    - Validate exclusively on Day 49 training samples (1.2k samples)
    - This matches test distribution
    - Validates day-shift generalization before submission

Expected R² Improvement Path:
  1. Validation R² (Day 49): 0.765 (measured)
  2. Leaderboard R² (Day 49, 41k samples): ~0.762 ± 0.01
     (slightly different from validation due to different sample distribution)
  3. If test includes Day 48 (unexpected): R² could spike to ~0.85
     (Indicates distribution shift detection in our validation caught real issue)

6.2 Core Module: feature_engineering.py
-----------------------------------------

Classes & Functions:

1. parse_timestamp_to_minutes(value: str) -> int
   Input: "H:M" format string (e.g., "14:30")
   Output: Minutes since midnight (e.g., 870)
   Validation: Checks hour [0, 23], minute [0, 59]
   Error handling: Raises ValueError on invalid format

2. decode_geohash(geohash: str) -> Tuple[float, float]
   Input: 6-character geohash (e.g., "qp02z1")
   Algorithm: Binary search in latitude/longitude space
     - For each character: Convert base32 → 5 bits
     - For each bit: Narrow lat/lon range
     - Even bits: Longitude, Odd bits: Latitude
   Output: Tuple (latitude, longitude) representing cell center
   Error handling: Validates base32 characters

3. decode_geohash_series(series: pd.Series) -> Tuple[pd.Series, pd.Series]
   Input: Pandas Series of geohash strings
   Optimization: Cache unique values to avoid redundant decoding
   Output: Two Series (latitudes, longitudes)
   Scalability: O(unique_geohashes) not O(total_rows)

4. class TargetEncoder
   Purpose: Convert high-cardinality categorical features to continuous
   
   Methods:
   
     __init__(cols, n_splits=5, smoothing=10.0, random_state=42):
       cols: List of column names to target-encode
       n_splits: K-Fold splits (5 for balance: train/val ratio)
       smoothing: Regularization parameter (10.0 standard)
       random_state: For reproducible K-Fold splits
   
     fit(features: pd.DataFrame, target: pd.Series) -> self:
       Fit encoder on entire dataset
       Output: Store global mean + per-category means
       Use case: Encoding test data (after training)
   
     transform(features: pd.DataFrame) -> pd.DataFrame:
       Map categorical values to learned means
       Output: New dataframe with columns {col_te, col_te, ...}
       Handles unseen categories: Map to global_mean_
   
     fit_transform(features: pd.DataFrame, target: pd.Series) -> pd.DataFrame:
       Fit encoder on training data, return OOF encoded values
       Process:
         1. K-Fold cross-validation loop (5 folds)
         2. For each fold: Fit on train folds, transform val fold
         3. Combine OOF predictions from all folds
         4. Final fit on entire dataset (for test transformation)
       Output: OOF-encoded training data (no leakage)
       Time: ~30 seconds for 77k rows

5. class FeatureBuilder
   @dataclass decorator: Automatically generates __init__, __repr__, etc.
   
   Attributes:
     - categorical_features: List of categorical column names
     - numerical_features: List of continuous column names
     - target_encoder: TargetEncoder instance (initialized in __post_init__)
     - fitted: Boolean flag (True after fit_transform)
   
   Methods:
   
     __post_init__():
       Initializes target_encoder with default parameters
       Called automatically after __init__
   
     _engineer_features(df: pd.DataFrame, fit_encoder: bool) -> pd.DataFrame:
       Internal method: Core feature engineering logic
       Steps:
         1. Parse timestamp → minutes → cyclical encoding (sin/cos)
         2. Create time_of_day categories
         3. Decode geohash → latitude/longitude
         4. Create hierarchical geohashes (2, 3, 4, 5 char)
         5. Create interaction features (roadtype_time, weather_time, geohash_time)
         6. Standardize categorical columns (fill missing)
       
       If fit_encoder=True: K-Fold target encoding (training)
       If fit_encoder=False: Apply fitted encoder (test)
       
       Returns: Full feature dataframe (~28 columns)
   
     fit_transform(X: pd.DataFrame, y: pd.Series) -> pd.DataFrame:
       Fit on training data, return transformed features
       Does feature engineering + K-Fold target encoding
       Output: Full training feature set (no leakage)
       Time: ~30-40 seconds
   
     transform(X: pd.DataFrame) -> pd.DataFrame:
       Apply feature engineering + fitted target encoding
       Use case: Transform test data
       Output: Test feature set (same columns as training)
       Time: ~5 seconds
     
     get_feature_names() -> List[str]:
       Returns: Ordered list of all 28 feature column names
       Use case: Selecting columns for model input

6.5 Metrics Module: metrics.py
-------------------------------

Functions:

  r2_score(y_true, y_pred) -> float:
    Computes coefficient of determination
    Formula: 1 - SS_res / SS_tot
    Range: [-inf, 1.0] (1.0 = perfect)
    Implementation: Uses sklearn.metrics.r2_score

  mean_absolute_error(y_true, y_pred) -> float:
    Computes average absolute error
    Formula: mean(|y_true - y_pred|)
    Interpretation: Avg prediction error (same units as target)
    For demand: 0.0399 ≈ 4% mean absolute deviation

  mean_squared_error(y_true, y_pred) -> float:
    Computes average squared error
    Formula: mean((y_true - y_pred)²)
    Penalizes large errors more

  regression_metrics(y_true, y_pred) -> Dict[str, float]:
    Convenience function: Computes all three metrics
    Returns: {'r2': ..., 'mae': ..., 'rmse': ...}

    - Decrease max_iter (400 → 200)

10.2 Known Limitations & Future Work
-------------------------------------

Limitations of Current Solution:

  1. Distribution Shift Awareness
     - Model trained primarily on Day 48 data
     - Test is entirely Day 49 (different distribution)
     - Mitigation: Validation strategy uses Day 49
     - Risk: If test has different weather/events, performance may vary

  2. Limited Weather Data
     - Temperature missing ~45% of values
     - Imputation by (geohash, weather) groups is heuristic
     - Ideal: Actual weather station data for interpolation
     - Impact: Temperature-based predictions less reliable

  3. Feature Sparsity in Sparse Locations
     - Small suburbs have few training samples
     - Target encoding smoothing may over-regularize
     - Ideal: Location-level cluster features
     - Impact: Predictions in sparse areas may have higher error

  4. No Temporal Trends
     - Model assumes static patterns across days
     - Reality: Traffic patterns may evolve (construction, events)
     - Ideal: Trend features (Day 1 vs Day 48 drift)
     - Impact: Longer-term predictions less reliable

  5. Cyclic Nature Not Fully Captured
     - Sin/cos encoding 24-hour cycle
     - Weekly cycles (Mon-Sun) not modeled
     - Ideal: Day-of-week features
     - Impact: Limited for multi-week predictions

Potential Improvements (Future Work):

  1. Ensemble Models
     - Combine HistGradient + CatBoost + LightGBM
     - Weighted average predictions
     - Expected gain: +0.5-1% R²

  2. Advanced Target Encoding
     - Hierarchical encoding (geohash level-by-level)
     - Interaction-aware encoding
     - Expected gain: +0.2-0.5% R²

  3. Feature Interactions
     - Explicit polynomial features (geohash × weather)
     - Learned embeddings for high-cardinality features
     - Expected gain: +0.3-0.8% R²

  4. Hyperparameter Optimization
     - Bayesian optimization (Optuna, Hyperopt)
     - Grid search over learning_rate × max_depth
     - Expected gain: +0.1-0.3% R²

  5. Outlier Detection
     - Remove anomalous training samples
     - Separate model for edge cases
     - Expected gain: +0.1-0.2% R²

10.3 Ethical Considerations & Bias
-----------------------------------

Potential Biases in Solution:

  1. Geographic Bias
     - Dataset only covers Bengaluru
     - Model cannot generalize to other cities
     - Mitigation: Acknowledge geographic scope

  2. Temporal Bias
     - Training on specific 2 days (Day 48-49)
     - May not represent typical weeks
     - Mitigation: Validate on Day 49 to detect shift

  3. Weather Dependency
     - Predictions assume weather patterns from training data
     - Unusual weather (heavy snow) not represented
     - Mitigation: Temperature range constraints

Responsible Use:
  - Model suitable for: Tactical traffic management (hours to days)
  - Model NOT suitable for: Long-term infrastructure planning (months/years)
  - Always pair with human expertise for critical decisions
  - Monitor model performance over time (real leaderboard updates)
